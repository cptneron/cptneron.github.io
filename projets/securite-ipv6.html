Exposition non filtrée
du NAS en IPv6

Un audit informel par un utilisateur de confiance a révélé que tous les ports du NAS étaient directement accessibles depuis Internet via IPv6 — sans aucune règle de firewall. Détection, analyse et durcissement en moins d'une heure.

📅 29 juin 2026 🖥️ NAS i5-3570 / Debian Bookworm 🔍 Découvert via pentest ami ⏱️ Corrigé en ~45 min
← Retour au projet NAS
Contexte

Comment la vulnérabilité a été découverte

Un ami ayant un accès utilisateur Jellyfin a effectué un scan Nmap sur l'IP publique du NAS. En IPv4, seul le port 32769 (Nginx Proxy Manager) est visible — la Freebox bloque tout le reste via NAT et ses règles de port forwarding.

Mais en IPv6, c'est une autre histoire. Sans NAT, le NAS est directement adressable depuis Internet, et ip6tables était en policy ACCEPT sans aucune règle.

⚠️ Situation avant correction Tous les services Docker (Radarr, Sonarr, Portainer, Transmission, OMV...) étaient accessibles depuis n'importe quelle adresse IPv6 sur Internet. L'adresse IPv6 publique du NAS était exposée directement sur l'interface physique enp4s0.

Correction

Mise en place des règles ip6tables

Avant — Firewall IPv6
Policy: ACCEPT
Règles: aucune

→ Tous les ports ouverts
→ Accessible depuis Internet
Après — Firewall IPv6
Policy: DROP
5 règles actives

→ 32769 ouvert (NPM)
→ 8322 ouvert (SSH)
→ Tout le reste bloqué
BASHRègles appliquées
# Policy par défaut : tout bloquer
ip6tables -P INPUT DROP

# Autoriser les connexions déjà établies
ip6tables -A INPUT -m state --state ESTABLISHED,RELATED -j ACCEPT

# Loopback
ip6tables -A INPUT -i lo -j ACCEPT

# Ports exposés volontairement
ip6tables -A INPUT -p tcp --dport 32769 -j ACCEPT  # NPM HTTPS
ip6tables -A INPUT -p tcp --dport 8322 -j ACCEPT   # SSH

# ICMPv6 nécessaire pour le fonctionnement IPv6
ip6tables -A INPUT -p ipv6-icmp -j ACCEPT
BASHVérification + persistance
$ ip6tables -L INPUT -n --line-numbers

Retour d'expérience

Leçons apprises

IPv6 change fondamentalement le modèle de sécurité. En IPv4 avec NAT, la box fait office de firewall implicite — seuls les ports explicitement forwardés sont accessibles. En IPv6, chaque machine est directement routable sur Internet. Ne pas avoir de règles ip6tables équivaut à ne pas avoir de firewall du tout.

Les outils de pentest sont des alliés. Sans le scan de cet ami, cette vulnérabilité aurait pu rester indétectée indéfiniment. Mettre en place des tests réguliers (Nmap, Shodan) fait partie d'une bonne hygiène d'administration.

La défense en profondeur reste importante. Même avec un firewall, chaque service exposé devrait avoir sa propre authentification. Fail2ban sur NPM et 2FA sur Jellyfin sont les prochaines étapes.
